# 消息队列对比 — 一篇文章看懂各种 MQ 的区别

---

## 一、什么是消息队列？

### 一句话

> **消息队列（Message Queue，MQ）** 就是一个「异步收件箱」—— 系统 A 把消息丢进去，系统 B 过一会儿再来取，双方不用同时在线。

### 没有 MQ 会怎样？

```
没有 MQ：         系统 A 直接调用系统 B
                  A ──────→ B
                  如果 B 挂了 → A 也崩了 💥
                  如果 B 慢   → A 一直等 ⏳

有了 MQ：         系统 A → [MQ] → 系统 B
                  A 丢完消息就走，不用等 B
                  B 挂了 → 消息在 MQ 里排队，B 恢复后继续处理
                  B 慢   → 消息堆积，不影响 A
```

### 三大核心作用

| 作用 | 说明 | 场景 |
|------|------|------|
| **解耦** | 系统之间不直接调用，通过 MQ 中转 | 订单系统 → MQ → 库存 / 物流 / 通知 |
| **削峰** | 瞬间高并发先存 MQ，后端慢慢消费 | 秒杀：10 万请求 → MQ → 后端按自己节奏处理 |
| **异步** | 耗时操作不阻塞主流程 | 注册后发邮件、短信验证码 |

## 二、主流消息队列一览

| 消息队列 | 出身 | 核心语言 | 定位 |
|----------|------|----------|------|
| **RabbitMQ** | 2007, Rabbit Technologies | Erlang | 通用消息中间件，AMQP 标准实现 |
| **Kafka** | 2011, LinkedIn → Apache | Java/Scala | 分布式流平台，海量日志/事件 |
| **RocketMQ** | 2012, 阿里 → Apache | Java | 金融级高可靠，阿里双十一验证 |
| **Pulsar** | 2012, Yahoo → Apache | Java | 下一代云原生消息平台 |
| **ActiveMQ** | 2004, Apache | Java | 老牌 JMS 消息中间件 |
| **Redis** | 2009, Salvatore Sanfilippo | C | 缓存/MQ 跨界选手 |
| **NSQ** | 2012, Bitly | Go | 极简分布式实时消息平台 |
| **ZeroMQ** | 2007, iMatix | C++ | 嵌入式消息库，不是完整的 MQ |

## 三、核心概念对比：三句话理解每个 MQ

### 3.1 RabbitMQ — 邮局

```
┌──────────┐         ┌──────────────┐         ┌──────────┐
│ 生产者 P  │ ────→  │  Exchange     │ ────→  │  队列 Q1  │ ────→ 消费者 C1
└──────────┘         │  (交换机)      │         └──────────┘
                     │  按规则分发    │         ┌──────────┐
                     │              │ ────→  │  队列 Q2  │ ────→ 消费者 C2
                     └──────────────┘         └──────────┘
```

**类比**：就像一个邮局，你寄信到 Exchange（分拣中心），它根据规则（routing key）把信投递到不同的 Queue（邮箱），收件人（消费者）从自己的邮箱取信。

**核心特点**：灵活的路由规则（Direct / Topic / Fanout / Headers），功能最全面的传统 MQ。

---

### 3.2 Kafka — 高速公路 + 行车记录仪

```
Producer ──→ ┌─────────────────────────────────────┐
             │  Topic: user-events （分区 0,1,2）    │
             │  ┌──────────────────────────────────┐│
             │  │ 消息 0, 1, 2, 3, 4, ... (有序)   ││
             │  └──────────────────────────────────┘│
             └─────────────────────────────────────┘ ──→ Consumer Group
```

**类比**：就像一条高速公路，消息按顺序写入、按顺序读取。消费者可以回放任意时刻的消息（就像行车记录仪回放），消息不会因为被消费了就删除。

**核心特点**：超高性能、消息持久化（磁盘顺序读写）、支持消息回溯（重放历史数据）。

---

### 3.3 RocketMQ — 阿里双十一的「快递员」

```
Producer ──→ ┌──────────────┐      ┌──────────────┐
             │  Broker 主    │ ──→  │  Broker 备    │  (主从同步，不丢消息)
             └──────────────┘      └──────────────┘
                    │
                    ▼
             Consumer (Push/Pull)
```

**类比**：阿里的快递员，经历了双十一的千锤百炼。消息必须送到，绝不丢件。支持事务消息（先下单再扣库存，有一个失败就回滚）。

**核心特点**：金融级可靠性（同步刷盘）、事务消息、延迟消息、万亿级消息堆积。

---

### 3.4 Pulsar — 模块化写字楼

```
┌──────────────────────────────────────────────┐
│  Pulsar                                      │
│  ┌────────────┐  ┌────────────┐              │
│  │ Broker 1   │  │ Broker 2   │  (无状态)    │
│  │ 计算层     │  │ 计算层     │              │
│  └─────┬──────┘  └─────┬──────┘              │
│        │               │                     │
│  ┌─────┴───────────────┴──────┐              │
│  │  BookKeeper (存储层)        │  (有状态)    │
│  │  分片存储，可独立扩缩容      │              │
│  └────────────────────────────┘              │
└──────────────────────────────────────────────┘
```

**类比**：模块化写字楼，计算层（Broker）和存储层（BookKeeper）分开，可以独立扩缩容。计算不够加 Broker，存储不够加 BookKeeper，互不影响。

**核心特点**：存算分离、多租户、原生支持跨地域复制（Geo-Replication）。

---

### 3.5 ActiveMQ — 老牌瑞士军刀

> 功能全面但性能一般，JMS 规范的标准实现，适合传统企业 Java 项目。

**核心特点**：支持多种协议（OpenWire、STOMP、AMQP、MQTT），插件丰富，但吞吐量是短板。

---

### 3.6 Redis Pub/Sub + Stream — 数据库顺手做 MQ

```
Pub/Sub 模式：             Stream 模式（Redis 5.0+）：
P ──→ Channel ──→ C1      P ──→ Stream ──→ Consumer Group
             ──→ C2                       ├─→ C1
                                          └─→ C2
```

**类比**：一个数据库顺手当了消息队列。Pub/Sub 像广播大喇叭（消息发完就没了，不在线收不到），Stream 像一个轻量级 Kafka（消息持久化）。

**核心特点**：部署简单（你大概率已经装了 Redis），适合轻量级场景，Pub/Sub 不持久化消息。

---

### 3.7 NSQ — 极简主义的 Go 语言 MQ

```
Producer ──→ nsqd ──→ Consumer
              │
              ▼
           nsqlookupd (服务发现)
```

**类比**：一个极简的分布式消息队列，Go 语言实现，单文件部署，没有复杂的配置，适合中小型项目。

**核心特点**：部署简单（一个二进制文件）、默认分布式、没有主节点（去中心化）。

## 六、怎么选？一张图搞定

```
你要做什么？
│
├── 日志收集、流处理、大数据管道
│   └── → Kafka（别无选择，它就是标准）
│
├── 金融交易、订单处理（绝不允许丢消息）
│   └── → RocketMQ（阿里双十一验证）或 Pulsar
│
├── 通用业务消息（微服务解耦、异步处理）
│   ├── 团队有 Java 经验 → RabbitMQ（生态最成熟）
│   └── 团队倾向轻量、Go → NSQ
│
├── 你已经用了 Redis，需求简单
│   └── → Redis Stream（别引入新组件）
│
├── IoT / 物联网（大量设备、低带宽）
│   └── → RabbitMQ (MQTT) 或 Pulsar (MQTT)
│
├── 传统企业 Java 项目（必须 JMS）
│   └── → ActiveMQ
│
└── 需要存算分离、云原生、多租户
    └── → Pulsar
```

## 七、一句话总结

| MQ | 一句话 |
|----|--------|
| **RabbitMQ** | 通用型，功能最全面，像邮局一样灵活 |
| **Kafka** | 大数据管道王者，吞吐量无敌，但运维复杂 |
| **RocketMQ** | 阿里出品，金融级可靠，经历过双十一的千锤百炼 |
| **Pulsar** | 下一代 MQ，存算分离，架构最先进 |
| **ActiveMQ** | 老牌 JMS，企业 Java 项目标配，但性能落后 |
| **Redis** | 轻量级跨界选手，简单场景够用，别当主力 MQ |
| **NSQ** | Go 语言极简 MQ，部署只要一个二进制文件 |
| **ZeroMQ** | 不是 MQ，是网络通信库，适合嵌入式场景 |

## 八、常见误区

| 误区 | 真相 |
|------|------|
| 「Kafka 最快，什么场景都用它」 | Kafka 适合大数据流，轻量业务消息用 RabbitMQ 延迟更低 |
| 「RabbitMQ 支持消息回溯」 | ❌ 不支持，消费了就没了（除非手动 ACK 后 requeue） |
| 「Redis Pub/Sub 可以当 MQ 用」 | Pub/Sub 消息不持久化，消费者离线就丢了，用 Stream 才行 |
| 「MQ 可以保证消息不丢失」 | 需要配置：持久化 + 手动 ACK + 集群 + 刷盘策略 |
| 「MQ 让系统变慢了」 | MQ 本身很快（微秒级），它让系统**响应更快**（异步） |
