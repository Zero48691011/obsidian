# DNS 域名解析系统详解

> 涵盖 DNS 核心原理、记录类型、解析流程、常用工具，以及主流云厂商的域名解析服务对比。

---

## 一、什么是 DNS？

**DNS（Domain Name System，域名系统）** 是互联网的「电话簿」。它把人能读懂的域名（如 `www.baidu.com`）翻译成机器能懂的 IP 地址（如 `110.242.68.66`）。

### 为什么需要 DNS？

- 人记域名容易，记 IP 难
- IP 地址可能变化（换服务器、CDN 调度），域名可以不变
- 一个域名可以对应多个 IP（负载均衡）
- 一个 IP 可以对应多个域名（虚拟主机）

## 二、DNS 的核心概念

### 2.1 域名层级结构

域名从右往左，层级从高到低：

```
www.example.com.
│   │       │   │
│   │       │   └── 根域（root，通常省略）
│   │       └────── 顶级域名（TLD，Top-Level Domain）
│   └────────────── 二级域名（SLD，Second-Level Domain）
└────────────────── 子域名（Subdomain）
```

- **根域**：`.` — 全球 13 组根服务器
- **顶级域名 (TLD)**：`.com`、`.cn`、`.org`、`.net` 等
- **二级域名**：`example.com`、`baidu.com`
- **子域名**：`www.example.com`、`api.example.com`

### 2.2 DNS 服务器类型

| 类型 | 说明 | 例子 |
|------|------|------|
| **根域名服务器** | 管理顶级域名服务器地址，全球 13 组 | `a.root-servers.net` ~ `m.root-servers.net` |
| **顶级域名服务器** | 管理某个 TLD 下的权威服务器 | `.com` 的 TLD 服务器 |
| **权威 DNS 服务器** | 存储某个域名的实际解析记录 | 阿里云 DNS、Cloudflare DNS |
| **递归 DNS 服务器** | 替客户端完成整个查询过程，并缓存结果 | `8.8.8.8`（Google）、`114.114.114.114`（国内） |

## 三、DNS 解析过程

### 3.1 完整解析流程

以浏览器访问 `www.example.com` 为例：

```
用户浏览器
    │
    ▼
本地 DNS 缓存（浏览器缓存 → OS hosts 文件 → 系统 DNS 缓存）
    │ 没命中
    ▼
递归 DNS 服务器（如 8.8.8.8）
    │ 没缓存
    ▼
根域名服务器 → 返回 .com 的 TLD 服务器地址
    │
    ▼
.com TLD 服务器 → 返回 example.com 的权威 DNS 服务器地址
    │
    ▼
example.com 权威 DNS 服务器 → 返回 www.example.com 的 IP
    │
    ▼
递归 DNS 服务器缓存结果 → 返回给浏览器
```

### 3.2 两种查询方式

| 查询方式 | 说明 | 谁用 |
|----------|------|------|
| **递归查询** | 客户端要求 DNS 服务器必须给出最终结果 | 客户端 → 递归 DNS |
| **迭代查询** | DNS 服务器只返回下一跳地址，让请求方自己继续查 | 递归 DNS → 根/TLD/权威 |

## 四、常见 DNS 记录类型

| 记录类型 | 用途 | 示例 |
|----------|------|------|
| **A** | 域名 → IPv4 地址 | `www.example.com → 93.184.216.34` |
| **AAAA** | 域名 → IPv6 地址 | `www.example.com → 2606:2800:220:1:248:1893:25c8:1946` |
| **CNAME** | 域名别名 → 另一个域名 | `www.example.com → example.com` |
| **MX** | 邮件交换记录，指定收件服务器 | `example.com → mail.example.com` (优先级 10) |
| **TXT** | 文本记录，常用于域名验证、SPF/DKIM | `example.com → "v=spf1 include:_spf.example.com ~all"` |
| **NS** | 指定该域名的权威 DNS 服务器 | `example.com → ns1.example.com` |
| **SOA** | 起始授权记录，域名的管理信息 | 包含主 DNS、管理员邮箱、序列号、刷新时间等 |
| **SRV** | 服务记录，指定特定服务的地址和端口 | `_sip._tcp.example.com → sip.example.com:5060` |
| **PTR** | 反向解析：IP → 域名 | `34.216.184.93.in-addr.arpa → www.example.com` |
| **CAA** | 证书颁发机构授权，限制谁能签发 SSL 证书 | `example.com → 0 issue "letsencrypt.org"` |

## 五、常用 DNS 诊断工具

### 5.1 dig（推荐）

```bash
# 基本查询
dig www.example.com

# 只显示结果（简洁模式）
dig www.example.com +short

# 指定 DNS 服务器查询
dig www.example.com @8.8.8.8

# 查询指定记录类型
dig example.com MX
dig example.com NS
dig example.com TXT

# 反向解析
dig -x 8.8.8.8

# 跟踪完整解析路径
dig www.example.com +trace
```

### 5.2 nslookup

```bash
# 基本查询
nslookup www.example.com

# 指定 DNS 服务器
nslookup www.example.com 8.8.8.8

# 交互模式
nslookup
> set type=mx
> example.com
```

### 5.3 host

```bash
# 基本查询
host www.example.com

# 查询 MX 记录
host -t MX example.com
```

### 5.4 在线工具

- [DNS Checker](https://dnschecker.org/) — 全球多节点 DNS 传播检测
- [What's My DNS](https://www.whatsmydns.net/) — 全球 DNS 传播检查
- [DNSPod 工具箱](https://tool.dnspod.cn/) — 腾讯云提供的 DNS 工具集

## 六、各云厂商 DNS 服务详解

### 6.1 AWS Route 53

**全称**：Amazon Route 53（53 指 DNS 端口号）

**核心功能**：

| 功能 | 说明 |
|------|------|
| 域名注册 | 支持注册数百种 TLD 域名 |
| DNS 解析 | 公共 DNS 和私有 DNS（VPC 内） |
| 健康检查 | 监控端点健康状态，自动故障转移 |
| 路由策略 | 简单路由、加权路由、延迟路由、故障转移、地理路由、多值应答 |
| DNSSEC | 支持域名签名验证 |

**路由策略详解**：

| 策略 | 场景 |
|------|------|
| **简单路由 (Simple)** | 一个域名 → 一个/多个 IP，随机返回 |
| **加权路由 (Weighted)** | 按权重比例分配流量（如 80% → 新版本，20% → 旧版本） |
| **延迟路由 (Latency)** | 自动解析到延迟最低的区域 |
| **故障转移 (Failover)** | 主备切换，主节点挂了自动切到备用 |
| **地理路由 (Geolocation)** | 根据用户地理位置返回不同 IP |
| **地理位置邻近 (Geoproximity)** | 基于用户和资源的地理位置，可设置偏置 |
| **多值应答 (Multivalue Answer)** | 随机返回多个健康 IP，客户端侧负载均衡 |

**常见应用场景**：
- 多区域部署 + 延迟路由 → 全球加速
- 加权路由 + 金丝雀发布 → 灰度上线
- 私有托管区 + VPC → 内部服务发现

**价格**（参考）：
- 托管区：$0.50/月/区
- 查询：$0.40/百万次（前 10 亿次）
- 健康检查：$0.50/月/个（基础）

## 八、DNS 最佳实践

### 8.1 TTL 设置建议

| 场景 | 推荐 TTL | 理由 |
|------|----------|------|
| **稳定服务** | 3600 - 86400 (1h-24h) | 减少 DNS 查询，提升解析速度 |
| **变更前** | 先降到 60-300 | 变更后快速生效 |
| **CDN / 故障转移** | 30 - 300 | 快速切换 |
| **负载均衡** | 60 - 600 | 平衡效果和性能 |

### 8.2 高可用设计

```
主 DNS（阿里云）  +  辅 DNS（DNSPod）
       │                    │
       └────────┬───────────┘
                │
           NS 记录配置两个
           ns1.example.com  → 阿里云
           ns2.example.com  → DNSPod
```

### 8.3 常见问题排查

| 问题 | 可能原因 | 排查命令 |
|------|----------|----------|
| 域名解析不了 | 域名未注册/过期、DNS 未配置 | `dig +trace domain.com` |
| 解析到旧 IP | TTL 缓存未过期、未修改成功 | `dig domain.com @8.8.8.8` 对比权威 |
| 部分地区解析异常 | DNS 劫持、运营商缓存 | 用全球 DNS 检测工具 |
| 邮件收不到 | MX 记录未配置或错误 | `dig domain.com MX` |
| HTTPS 证书错误 | CNAME 指向的域名和证书不匹配 | `dig domain.com` 确认解析结果 |

### 8.4 安全建议

- **开启 DNSSEC**：防止 DNS 劫持和缓存投毒
- **使用 DNS over HTTPS (DoH) / DNS over TLS (DoT)**：加密 DNS 查询
- **配置 CAA 记录**：限制 SSL 证书颁发机构
- **定期审计 DNS 记录**：清理不再使用的记录
- **监控 DNS 变更**：开启变更告警通知

## 十、总结

1. **选型建议**：
   - **国内业务为主** → 阿里云 DNS 或 DNSPod，智能解析分运营商/地域效果好
   - **全球业务** → Cloudflare（免费且强大）或 Route 53（与 AWS 生态集成）
   - **云原生/VPC 内** → 首选对应云厂商的私有 DNS

2. **省钱技巧**：
   - Cloudflare 免费版已足够大多数个人/小型网站
   - 国内厂商免费版也够用，需要智能解析再升级

3. **关键原则**：
   - 变更前先降 TTL
   - 主备 DNS 分不同厂商
   - 开启 DNSSEC 和变更告警
