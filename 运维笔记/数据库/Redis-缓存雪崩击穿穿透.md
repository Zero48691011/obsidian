# Redis 缓存雪崩、击穿、穿透 详解

---

## 一、先搞清楚三个概念

这三个词长得像，但原因和解决方案完全不同：

| 问题 | 一句话 | 关键词 |
|------|--------|--------|
| **缓存雪崩** | 大量缓存**同时过期**，请求瞬间全砸到数据库 | 同一时刻、大面积失效 |
| **缓存击穿** | 一个**热点 key** 过期，大量并发请求砸向数据库 | 单个热点 key、高并发 |
| **缓存穿透** | 查一个**数据库里也没有**的数据，缓存形同虚设 | 不存在的数据、恶意攻击 |

---

## 二、缓存雪崩（Cache Avalanche）

### 2.1 官方定义

> **缓存雪崩**是指缓存中大量的 key 在**同一时间点过期**，导致所有请求直接打到数据库，数据库瞬间压力过大，可能直接宕机，进而引发整个系统的连锁崩溃。

### 2.2 通俗解释

#### 类比：超市收银台

```
正常情况：
  收银台（缓存）有 10 个窗口，顾客（请求）排队结账，很快。
  偶尔有顾客找不到商品，去仓库（数据库）查一下，不碍事。

雪崩发生时：
  10 个收银员同时下班（缓存集体过期），顾客全部涌向仓库管理员（数据库）
  一个人查库存 → 仓库管理员：「来了来了！」
  100 人同时查库存 → 仓库管理员：「...」→ 当场累垮 → 超市瘫痪
```

### 2.3 发生过程

```
时间线：
  00:00  缓存中 10000 个 key，TTL 统一设为 3600 秒
  00:59  一切正常，QPS 5000，数据库负载 20%
  01:00  10000 个 key 同时过期！缓存瞬间空了
  01:00  所有请求打到数据库，数据库 QPS 从 500 飙到 50000
  01:01  数据库连接池耗尽、CPU 100%、响应超时
  01:02  应用层超时重试 → 更多请求打到数据库 → 恶性循环
  01:05  数据库宕机，整个系统不可用
```

```
        请求量
          ▲
    50000 │                    ● ← 雪崩！数据库被打爆
          │                   ╱ ╲
          │                  ╱   ╲
          │                 ╱     ╲
          │                ╱       ╲
          │               ╱         ╲________
     5000 │──────────────╱                   ╲______
          │             ╱
          └──────────────────────────────────────────▶ 时间
               缓存正常期  │← 雪崩 →│  恢复期（如果还能恢复）
                          │
                    大量 key 同时过期
```

### 2.4 常见原因

| 原因 | 说明 |
|------|------|
| **同一时间批量写入缓存** | 定时任务在 00:00 全量刷新缓存，所有 key 的 TTL 相同 |
| **TTL 设置相同** | 业务代码里所有 key 都设了 3600 秒过期 |
| **缓存服务器重启** | Redis 宕机/重启后，所有数据丢失，雪崩式重建 |
| **热点数据集中过期** | 秒杀活动开始前一次性预热缓存，活动结束后集体过期 |

## 三、缓存击穿（Cache Breakdown）

### 3.1 一句话

> 某个**超级热点 key**（比如秒杀商品的库存）过期了，瞬间大量并发请求同时去查数据库，把数据库打穿。

### 3.2 和雪崩的区别

```
雪崩：大量 key 同时过期 → 大面积故障
       ┌──┐┌──┐┌──┐┌──┐┌──┐  ← 10000 个 key 同时过期
       └──┘└──┘└──┘└──┘└──┘

击穿：一个热点 key 过期 → 单点故障
       ┌──┐                    ← 就这一个 key 过期
       └──┘                    ← 但这一个 key 扛了 80% 的流量
```

### 3.3 典型场景

```
秒杀场景：
  key: "seckill:stock:iphone16"  ← 100 万人盯着这个 key
  TTL: 60 秒

  第 59 秒：缓存里有库存数据，Redis 扛住 10 万 QPS
  第 60 秒：key 过期
  第 61 秒：10 万 QPS 全部打到 MySQL → MySQL 瞬间崩溃
```

---

## 四、缓存穿透（Cache Penetration）

### 4.1 一句话

> 查询一个**数据库里根本不存在**的数据，缓存永远用不上，每次请求都穿透到数据库。

### 4.2 典型场景

```
恶意攻击：
  攻击者：curl http://api.com/product/-1
  缓存：查 product:-1 → 没有
  数据库：查 product:-1 → 也没有
  缓存：没查到所以不缓存
  攻击者：curl http://api.com/product/-1  ← 再发 100 万次
  数据库：每次都查，每次都查不到，但每次都扛了请求

结果：数据库被不存在的 ID 打爆
```

### 4.3 三个问题对比

```
请求 ──▶ 缓存 ──▶ 有数据？──▶ 返回 ✅
                │
                └──▶ 没数据 ──▶ 查数据库 ──▶ 有数据？──▶ 更新缓存 → 返回 ✅
                                                   │
                                                   └──▶ 没数据 → 返回空 ❌
                                                         ↑
                                                   穿透发生在这里！
                                                   每次查不存在的 ID 都走到这里
```

## 五、解决方案

### 5.1 雪崩 — 解决方案

#### 方案 A：过期时间加随机值（最简单）

```python
import random

# ❌ 错误写法
redis.setex("user:123", 3600, data)           # 所有 key 都是 3600 秒

# ✅ 正确写法
base_ttl = 3600
random_ttl = random.randint(0, 600)            # 随机 0~600 秒
redis.setex("user:123", base_ttl + random_ttl, data)  # 3600~4200 秒随机
```

**效果**：10000 个 key 的过期时间分散在 3600~4200 秒之间，不会同时过期。

#### 方案 B：多级缓存（本地缓存 + Redis）

```
请求 ──▶ 本地缓存 (Caffeine/Guava) ──▶ 命中？──▶ 返回 ✅
                      │
                      └──▶ 未命中 ──▶ Redis ──▶ 命中？──▶ 返回 ✅
                                                │
                                                └──▶ 未命中 ──▶ 数据库
```

**效果**：即使 Redis 挂了，本地缓存还能扛一阵，给数据库缓冲时间。

#### 方案 C：缓存不过期 + 异步更新

```python
# ❌ 错误做法：设置固定过期时间
redis.setex("hot:data", 3600, data)

# ✅ 正确做法：逻辑过期，值里带过期时间戳
cache_data = {
    "data": data,
    "expire_at": time.time() + 3600    # 逻辑过期时间
}
redis.set("hot:data", json.dumps(cache_data))  # 不设物理过期

# 读取时：
def get_data(key):
    raw = redis.get(key)
    if raw is None:
        return load_from_db(key)       # 真没了才查数据库
    cache = json.loads(raw)
    if time.time() > cache["expire_at"]:
        # 逻辑过期了，异步更新，但当前请求先用旧数据返回
        async_refresh(key)             # 后台线程去更新
    return cache["data"]
```

**效果**：缓存永不过期（物理上），永远有数据可返回，数据库压力可控。

#### 方案 D：Redis 高可用（哨兵 / 集群）

```
┌──────────────┐
│    Sentinel   │  ← 监控 Redis 主节点
└──────┬───────┘
       │
┌──────┴──────┐      ┌──────────────┐
│ Redis Master │─────▶│ Redis Slave  │  ← 主挂了，自动切换
└──────────────┘      └──────────────┘
```

**效果**：Redis 本身不会因为单点故障而全挂。

#### 方案 E：限流降级

```java
// 使用 Sentinel / Hystrix 限流
@SentinelResource(value = "getUser", fallback = "getUserFallback")
public User getUser(Long id) {
    // 先查缓存，再查数据库
}

// 降级方法：数据库扛不住时返回兜底数据
public User getUserFallback(Long id) {
    return new User(id, "默认用户", "系统繁忙，请稍后再试");
}
```

**效果**：数据库被打爆时，至少还能返回一个降级结果，而不是直接 500。

### 5.2 击穿 — 解决方案

#### 方案 A：互斥锁（Mutex Lock）

```python
def get_data_with_lock(key):
    # 1. 先查缓存
    data = redis.get(key)
    if data is not None:
        return data

    # 2. 缓存未命中，尝试获取锁
    lock_key = f"lock:{key}"
    if redis.setnx(lock_key, 1, ex=10):  # 获取锁，10 秒自动释放
        try:
            # 3. 双重检查：拿到锁后再查一次缓存
            data = redis.get(key)
            if data is not None:
                return data

            # 4. 查数据库
            data = db.query("SELECT * FROM ...")
            redis.setex(key, 3600, data)
            return data
        finally:
            redis.delete(lock_key)  # 释放锁
    else:
        # 5. 没拿到锁，等一会儿再重试
        time.sleep(0.1)
        return get_data_with_lock(key)  # 递归重试
```

**效果**：100 个并发请求同时发现缓存过期，只有 1 个去查数据库，其余 99 个自旋等待。

**流程图**：

```
并发请求 × 1000
    │
    ├─ 请求 1 ──▶ 缓存未命中 ──▶ 获得锁 ✅ ──▶ 查数据库 ──▶ 更新缓存 ──▶ 释放锁
    │
    ├─ 请求 2 ──▶ 缓存未命中 ──▶ 获得锁 ❌ ──▶ 等待 100ms ──▶ 重试 ──▶ 缓存命中 ✅
    │
    ├─ 请求 3 ──▶ 缓存未命中 ──▶ 获得锁 ❌ ──▶ 等待 100ms ──▶ 重试 ──▶ 缓存命中 ✅
    │
    └─ ... 其余 997 个同理
```

#### 方案 B：逻辑过期（永不过期 + 异步刷新）

和雪崩的「方案 C」相同，对于热点 key 特别有效：

```python
def get_hot_data(key):
    raw = redis.get(key)
    cache = json.loads(raw)

    if time.time() > cache["expire_at"]:
        # 逻辑过期 → 异步刷新，当前请求先用旧数据
        thread_pool.submit(refresh_cache, key)
    return cache["data"]   # 哪怕过期了也先返回旧数据
```

#### 方案 C：永远不过期（对热点 key）

```python
# 秒杀商品的库存，直接不设过期时间
redis.set("seckill:stock:iphone16", 1000)  # 不设 TTL

# 库存变更时主动更新
def reduce_stock(product_id):
    redis.decr(f"seckill:stock:{product_id}")
    # 同步更新数据库（异步消息队列）
    mq.send({"action": "reduce_stock", "product_id": product_id})
```

### 5.3 穿透 — 解决方案

#### 方案 A：缓存空值

```python
def get_data(key):
    data = redis.get(key)
    if data is not None:
        if data == "__NULL__":
            return None          # 缓存了空值，直接返回
        return data

    data = db.query(...)
    if data is None:
        redis.setex(key, 60, "__NULL__")  # 缓存空值，TTL 短一些
        return None
    else:
        redis.setex(key, 3600, data)
        return data
```

**效果**：第一次查 `product:-1` 会发现数据库里没有，缓存一个 `__NULL__`。接下来 60 秒内再查 `product:-1`，直接从缓存返回空，不再穿透到数据库。

#### 方案 B：布隆过滤器（Bloom Filter）

```
布隆过滤器原理：
  一个位数组 + 多个哈希函数，提前把所有「存在的数据 ID」放进去。

  请求：product:99999
    → 布隆过滤器判断：这个 ID 在集合里吗？
       ├─ 可能存在 → 放行，查缓存/数据库
       └─ 一定不存在 → 直接拦截，返回空，不查数据库
```

```python
from pybloom_live import BloomFilter

# 初始化：预计 100 万条数据，误判率 0.1%
bloom = BloomFilter(capacity=1000000, error_rate=0.001)

# 启动时加载所有已有 ID
for product_id in db.query("SELECT id FROM products"):
    bloom.add(product_id)

# 查询时：
def get_product(product_id):
    if product_id not in bloom:   # 一定不存在
        return None                # 直接返回，不查数据库
    # 可能存在，继续查缓存 → 数据库
    return get_from_cache_or_db(product_id)
```

**效果**：99.9% 的非法 ID 被布隆过滤器直接拦截，连缓存都不用查。

#### 方案 C：参数校验

```python
# 最简单但最有效的第一道防线
def get_product(product_id):
    if product_id <= 0 or product_id > 10000000:
        raise ValueError("非法商品 ID")
    # 继续正常查询...
```

**效果**：在业务层就拦截掉明显非法的请求，不浪费任何后端资源。

## 六、完整防护体系

把三种方案组合起来，构成一个完整的缓存防护架构：

```
请求 ──▶ 参数校验 ──▶ 非法？──▶ 拒绝 ❌
              │
              ▼
        布隆过滤器 ──▶ 一定不存在？──▶ 返回空 ❌
              │
              ▼
        本地缓存 ──▶ 命中？──▶ 返回 ✅
              │
              ▼
          Redis ──▶ 命中？──▶ 返回 ✅
              │
              ▼
        获取互斥锁 ──▶ 没拿到锁？──▶ 等待重试
              │
              ▼
         查数据库 ──▶ 有数据？──▶ 更新缓存 → 返回 ✅
              │
              ▼
        缓存空值 ──▶ 返回空 ❌
```

| 层级 | 防御什么 | 方案 |
|------|---------|------|
| 1. 参数校验 | 穿透 | 拦截明显非法的 ID |
| 2. 布隆过滤器 | 穿透 | 拦截不存在的数据（误判率 0.1%） |
| 3. 本地缓存 | 雪崩 | Redis 挂了也能扛一阵 |
| 4. Redis + 随机 TTL | 雪崩 | 避免同时过期 |
| 5. 互斥锁 | 击穿 | 热点 key 过期时只有 1 个请求查库 |
| 6. 缓存空值 | 穿透 | 不存在的 key 也缓存 |
| 7. 限流降级 | 雪崩 | 数据库扛不住时返回兜底数据 |

## 七、面试怎么答

**面试官**：「说说 Redis 缓存雪崩、击穿、穿透是什么，怎么解决？」

**推荐回答结构**：

```
1. 先说区别（30 秒）：
   「雪崩是大量 key 同时过期，击穿是单个热点 key 过期，
    穿透是查数据库里根本不存在的数据。」

2. 分别说原因 + 方案（2 分钟）：
   雪崩 → 随机 TTL、多级缓存、高可用 → 「核心是让过期时间分散」
   击穿 → 互斥锁、逻辑过期 → 「核心是让查库的行为串行化」
   穿透 → 布隆过滤器、缓存空值 → 「核心是让不存在的 key 也有缓存」

3. 总结一句话（10 秒）：
   「雪崩防大面积，击穿防单点，穿透防空查询。」
```

## 八、总结

| 问题 | 原因 | 核心方案 | 一句话 |
|------|------|---------|--------|
| **雪崩** | 大量 key 同时过期 | 随机 TTL + 多级缓存 + 高可用 | 让过期时间「错开」 |
| **击穿** | 热点 key 过期 + 高并发 | 互斥锁 + 逻辑过期 | 让查库行为「排队」 |
| **穿透** | 查不存在的数据 | 布隆过滤器 + 缓存空值 + 参数校验 | 让空值「有缓存」 |

> **核心思想**：缓存是数据库的「防弹衣」，雪崩/击穿/穿透就是防弹衣被撕开的口子。解决方案的思路不是让防弹衣永远不破，而是让破了之后子弹（请求）不会全部打到身体（数据库）上。
