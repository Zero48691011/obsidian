# Redis 原理详解

---

## 一、一句话概括

> **Redis 是一个基于内存的单线程键值数据库，通过 I/O 多路复用实现高性能，用「单线程 + 事件驱动」模型避免了锁竞争，用多种底层数据结构支撑丰富的上层数据类型。**

---

## 二、为什么 Redis 这么快？

### 2.1 官方数据

| 指标 | 数值 |
|------|------|
| 读写速度 | 10 万+ QPS（单机） |
| 延迟 | 亚毫秒级（< 1ms） |
| 核心原因 | 纯内存操作 + 单线程无锁 + I/O 多路复用 |

### 2.2 快的三大支柱

```
Redis 高性能
    │
    ├── 1. 纯内存操作
    │     所有数据在内存中，读写不涉及磁盘 I/O
    │     对比 MySQL：一次查询 = 磁盘寻道 + 磁盘读取 + 网络传输
    │
    ├── 2. 单线程模型（核心）
    │     单线程处理所有命令，没有锁竞争、没有上下文切换开销
    │     对比多线程：省去了互斥锁、信号量、CAS 的开销
    │
    └── 3. I/O 多路复用
          一个线程同时监听多个 socket，哪个有数据就处理哪个
          对比传统 BIO：不需要为每个连接创建一个线程
```

### 2.3 单线程 ≠ 只有一个线程

```
Redis 的线程分工（Redis 6.0+）：

主线程          ← 处理命令（单线程，核心）
    │
    ├── 后台线程 1 → 处理 AOF 刷盘
    ├── 后台线程 2 → 处理 RDB 快照（fork 子进程）
    ├── 后台线程 3 → 异步删除大 key（UNLINK）
    └── IO 线程（6.0+）→ 网络读写（多线程，但命令执行仍是单线程）
```

**重点**：执行命令 `SET`、`GET`、`INCR` 等操作的核心线程只有一个，这是 Redis 设计的精髓。

## 三、单线程模型详解

### 3.1 事件驱动架构

```
Redis 主线程就是一个死循环，不断处理事件：

while (true) {
    // 1. 等待事件（阻塞，直到有事件发生）
    events = aeProcessEvents()
    
    // 2. 处理文件事件（socket 可读/可写）
    for (event in events.file_events) {
        handleFileEvent(event)
    }
    
    // 3. 处理时间事件（定时任务：过期 key 清理、AOF 重写等）
    for (event in events.time_events) {
        handleTimeEvent(event)
    }
}
```

### 3.2 I/O 多路复用

这就是 Redis 单线程能扛 10 万并发的关键：

```
传统 BIO 模型（一个连接一个线程）：
  Client 1 ──▶ Thread 1 ──▶ Redis
  Client 2 ──▶ Thread 2 ──▶ Redis
  Client 3 ──▶ Thread 3 ──▶ Redis
  ...
  Client 10000 ──▶ Thread 10000 → 内存爆炸，上下文切换开销巨大

Redis I/O 多路复用（一个线程监听所有连接）：
  Client 1 ──┐
  Client 2 ──┤
  Client 3 ──┼──▶ epoll/select/kqueue ──▶ Redis 主线程
  Client 4 ──┤    （操作系统内核事件通知）
  Client 5 ──┘
     ↑
  只有有数据到达的连接才会被处理，空闲连接不消耗 CPU
```

```
内核通知流程：

1. Redis 调用 epoll_wait() 阻塞等待
2. Client 1 发送数据 → 内核收到数据 → 通知 epoll
3. epoll_wait() 返回 → 告诉 Redis：「Client 1 可读了」
4. Redis 读取 Client 1 的数据，执行命令，返回结果
5. Redis 再次调用 epoll_wait() 阻塞等待
```

### 3.3 为什么单线程反而更快？

| 对比维度 | 多线程 | Redis 单线程 |
|---------|--------|-------------|
| **锁开销** | 读写共享数据需要加锁，锁竞争浪费 CPU | 无锁，数据天然线程安全 |
| **上下文切换** | 线程切换需要保存/恢复寄存器、栈，开销大 | 无切换开销 |
| **并发控制** | 需要 Mutex、CAS、原子操作 | 不需要，所有操作都是原子的 |
| **CPU 瓶颈** | 真正的瓶颈是内存和网络，不是 CPU | 单线程 CPU 足够用 |

**一句话**：Redis 的瓶颈从来不是 CPU，而是**内存带宽**和**网络 I/O**。单线程足够快，多线程反而会引入锁开销。

## 四、底层数据结构

Redis 的每种上层数据类型，底层都由一种或多种**高效数据结构**实现：

### 4.1 对照表

| 上层类型 | 底层实现（条件切换） | 时间复杂度 |
|---------|-------------------|-----------|
| **String** | SDS（简单动态字符串） | O(1) |
| **List** | QuickList（3.2+）= 双向链表 + 压缩列表 | O(1) 头尾操作 |
| **Hash** | ListPack（小数据）/ HashTable（大数据） | O(1) |
| **Set** | IntSet（纯整数）/ HashTable | O(1) |
| **ZSet** | ListPack（小数据）/ SkipList + HashTable | O(log N) |
| **Stream** | Rax（基数树） | O(log N) |

### 4.2 SDS（Simple Dynamic String）— 为什么不用 C 字符串？

```c
// C 语言的字符串获取长度需要 O(n) 遍历
strlen("hello")  → 遍历直到 '\0'，5 个字节 → 5 次循环

// SDS 存储了长度，O(1) 即可获取
struct sdshdr {
    int len;     // 已用长度 → 直接读，O(1)
    int free;    // 剩余空间
    char buf[];  // 实际数据
};
```

| 对比 | C 字符串 | SDS |
|------|---------|-----|
| 获取长度 | O(n) 遍历 | O(1) 读 len |
| 缓冲区溢出 | 容易（不记录容量） | 安全（自动扩容） |
| 内存分配 | 每次修改都重新分配 | 预分配 + 惰性释放 |
| 二进制安全 | ❌ 遇 `\0` 截断 | ✅ 存任意二进制数据 |

### 4.3 SkipList（跳表）— ZSet 的核心

```
跳表 = 多层索引的有序链表，查找时跳过大量节点

Level 3:   1 ──────────────────────── 9 ──────── 12
Level 2:   1 ──────── 5 ──────────── 9 ──────── 12
Level 1:   1 ── 3 ── 5 ── 7 ──────── 9 ── 11 ─ 12
Level 0:   1 ─ 3 ─ 5 ─ 7 ─ 8 ─ 9 ─ 10 ─ 11 ─ 12

查找 11：
  从 Level 3 开始：1 → 9 → 12（12 > 11，退回 9）
  下降到 Level 2：9 → 12（12 > 11，退回 9）
  下降到 Level 1：9 → 11（找到了！）
  只需 6 次比较，而遍历 Level 0 需要 9 次
```

| 对比 | 跳表 | 平衡树（红黑树/AVL） |
|------|------|-------------------|
| 查找 | O(log N) | O(log N) |
| 插入 | O(log N)，实现简单 | O(log N)，需要旋转 |
| 范围查询 | ✅ 极快（链表天然有序） | 需要中序遍历 |
| 实现复杂度 | 简单 | 复杂 |

**Redis 选跳表而不选红黑树**：跳表实现简单、支持范围查询、插入删除不需要旋转再平衡。

## 五、数据过期机制

### 5.1 两种策略组合

```
Redis 不是到时间了立刻删除，而是「惰性删除 + 定期删除」配合：

策略 1：惰性删除（Lazy Expiration）
  访问 key 时检查 → 过期了 → 返回空 + 删除
  ✅ 对 CPU 友好（不主动扫描）
  ❌ 没人访问的过期 key 永远占着内存

策略 2：定期删除（Active Expiration）
  每 100ms 随机抽 20 个 key → 删除其中过期的
  → 如果过期比例 > 25%，继续循环
  ✅ 清理「冷数据」过期 key
  ❌ 不能太频繁，否则浪费 CPU
```

### 5.2 内存淘汰策略（满了怎么办？）

```
当内存达到 maxmemory 时，Redis 按以下策略淘汰 key：

┌─────────────────────────────────────────────────────┐
│  策略                    │  行为                      │
├─────────────────────────────────────────────────────┤
│  noeviction             │  拒绝写入，只读不删（默认）    │
│  allkeys-lru            │  所有 key 中，淘汰最近最少用的  │
│  volatile-lru           │  有过期时间的 key 中，淘汰 LRU  │
│  allkeys-lfu            │  所有 key 中，淘汰最不常用的    │
│  volatile-lfu           │  有过期时间的 key 中，淘汰 LFU  │
│  allkeys-random         │  所有 key 中随机淘汰           │
│  volatile-random        │  有过期时间的 key 中随机淘汰    │
│  volatile-ttl           │  有过期时间的 key 中，淘汰 TTL 最短的 │
└─────────────────────────────────────────────────────┘
```

**推荐**：缓存场景用 `allkeys-lru`，消息队列场景用 `noeviction`。

## 六、持久化

Redis 是内存数据库，断电数据就没了，所以需要持久化：

### 6.1 RDB（Redis Database）— 快照

```
工作原理：
  1. Redis 调用 fork() 创建子进程
  2. 父进程继续处理请求，子进程把当前内存数据写到磁盘
  3. 写完后替换旧的 RDB 文件

触发方式：
  手动：SAVE（阻塞）/ BGSAVE（后台）
  自动：save 900 1    ← 900 秒内至少 1 次修改

优点：文件小、恢复快、对性能影响小（子进程写）
缺点：两次快照之间的数据会丢失（秒级/分钟级）
```

```
时间线：
  T0: 最后一次快照 ────────────────── T1: 宕机
       ↑                                ↑
    数据完整                    这之间的数据丢失了！
```

### 6.2 AOF（Append Only File）— 日志

```
工作原理：
  1. 每条写命令都追加到 AOF 文件末尾
  2. 重启时重放 AOF 文件中的命令，重建数据

刷盘策略：
  appendfsync always   → 每条命令都刷盘（最安全，最慢）
  appendfsync everysec → 每秒刷一次（推荐，丢 1 秒数据）
  appendfsync no       → 由操作系统决定（最快，最不安全）

AOF 重写（压缩）：
  原始 AOF：SET counter 1 → SET counter 2 → SET counter 3
  重写后：  SET counter 3   ← 中间状态被压缩掉了
```

### 6.3 RDB vs AOF

| 维度 | RDB | AOF |
|------|-----|-----|
| 原理 | 内存快照 | 命令日志 |
| 文件大小 | 小 | 大（可重写压缩） |
| 恢复速度 | 快 | 慢（逐条重放） |
| 数据安全 | 丢几分钟数据 | 最多丢 1 秒 |
| 性能影响 | fork 时短暂阻塞 | 持续 I/O 开销 |
| **推荐** | 两者都开，AOF 优先 | — |

### 6.4 混合持久化（Redis 4.0+）

```
RDB + AOF 混合：
  重写 AOF 时，前半部分用 RDB 二进制快照，后半部分追加增量命令

  文件结构：
  ┌──────────────────────┬──────────────────────┐
  │  RDB 快照（二进制）      │  AOF 增量命令（文本）    │
  └──────────────────────┴──────────────────────┘
  
  恢复时：先加载 RDB 部分（快），再重放增量的 AOF（数据完整）
```

## 七、主从复制

### 7.1 工作原理

```
复制流程（全量 + 增量）：

  1. 从节点连接主节点，发送 PSYNC 命令
  2. 主节点执行 BGSAVE，生成 RDB 快照
  3. 主节点把 RDB 发给从节点，同时把新命令写入 replication buffer
  4. 从节点加载 RDB，然后主节点把 buffer 中的增量命令发给从节点
  5. 之后进入增量同步：主节点每条写命令都异步发给从节点

┌──────────────┐     复制      ┌──────────────┐
│  Redis Master │ ──────────▶  │  Redis Slave  │
│  读写         │              │  只读          │
└──────────────┘              └──────────────┘
```

### 7.2 复制偏移量

```
主节点和从节点各自维护一个 replication offset：

  主节点 offset: 1000
  从节点 offset: 950  ← 差了 50 条命令没同步

  断线重连后：
  - 如果偏移量差别不大（在 replication backlog 范围内）→ 增量同步
  - 如果偏移量差别太大（backlog 不够）→ 全量同步
```

### 7.3 哨兵（Sentinel）— 高可用

```
┌────────────────────────────────────────────┐
│              Sentinel 集群                   │
│  Sentinel 1  Sentinel 2  Sentinel 3        │
│       │            │            │           │
│       └────────────┼────────────┘           │
│                    │ 监控                    │
│       ┌────────────┴────────────┐           │
│       │      Redis Master       │           │
│       └────────────┬────────────┘           │
│                    │ 复制                    │
│       ┌────────────┴────────────┐           │
│       │      Redis Slave        │           │
│       └─────────────────────────┘           │
│                                             │
│  主节点挂了 → Sentinel 投票选新主 → 自动切换   │
└────────────────────────────────────────────┘
```

## 八、集群模式（Redis Cluster）

### 8.1 数据分片：哈希槽（Hash Slot）

```
Redis Cluster 把数据分成 16384 个槽（slot）：

  slot = CRC16(key) % 16384

  3 节点集群：
  Node 1: 槽 0 ~ 5460
  Node 2: 槽 5461 ~ 10922
  Node 3: 槽 10923 ~ 16383
```

```
请求重定向：

  Client: GET user:1000
  → 计算 slot = CRC16("user:1000") % 16384 = 8000
  → 请求发到 Node 1
  → Node 1 返回 MOVED 8000 192.168.1.3:6379
  → Client 重新连接 Node 3，再发请求
```

### 8.2 集群 vs 哨兵

| 维度 | 哨兵模式 | 集群模式 |
|------|---------|---------|
| 数据分片 | ❌ 每个节点存全部数据 | ✅ 数据分散到多个节点 |
| 水平扩展 | ❌ 只能加从节点（读扩展） | ✅ 加节点自动分片 |
| 高可用 | ✅ 自动故障转移 | ✅ 自动故障转移 + 副本 |
| 复杂度 | 低 | 高 |
| 适用场景 | 数据量 < 内存，追求简单 | 数据量大，需要水平扩展 |

## 九、Redis 6.0+ 多线程 I/O

Redis 6.0 引入了多线程 I/O，但**命令执行仍然是单线程**：

```
Redis 6.0 之前：
  主线程：读网络 → 解析命令 → 执行命令 → 写网络（全部单线程）
           ↑ 网络 I/O 是瓶颈

Redis 6.0 之后：
  IO 线程 1：读网络 → 解析命令 ──┐
  IO 线程 2：读网络 → 解析命令 ──┤
  IO 线程 3：读网络 → 解析命令 ──┼──▶ 主线程：执行命令（仍然是单线程！）
  IO 线程 4：读网络 → 解析命令 ──┘
                                 │
  IO 线程 1：写网络 ← 主线程分派 ←┘
  IO 线程 2：写网络
  IO 线程 3：写网络
  IO 线程 4：写网络
```

**默认关闭**，需要手动开启：

```conf
io-threads 4                  # 开启 4 个 IO 线程
io-threads-do-reads yes       # 读也启用多线程
```

## 十、总结

| 维度 | 核心原理 |
|------|---------|
| **为什么快** | 内存操作 + 单线程无锁 + I/O 多路复用（epoll） |
| **线程模型** | 主线程单线程执行命令，后台线程处理持久化/异步删除，6.0+ IO 多线程 |
| **数据结构** | SDS、SkipList、QuickList、ListPack、HashTable，每种都为特定场景优化 |
| **过期策略** | 惰性删除（访问时检查）+ 定期删除（100ms 随机采样） |
| **淘汰策略** | 8 种，推荐缓存用 allkeys-lru，消息队列用 noeviction |
| **持久化** | RDB（快照，快但丢数据）+ AOF（日志，安全但慢）+ 混合（4.0+） |
| **高可用** | 主从复制 + 哨兵自动故障转移 |
| **集群** | 16384 个哈希槽分片，数据自动分布 |
| **原子性** | 单线程模型天然保证每个命令的原子性，Lua 脚本保证多命令原子性 |

> **一句话总结**：Redis 的核心设计哲学是「简单即高效」—— 用单线程规避锁竞争，用高效数据结构替代复杂算法，用事件驱动模型替代多线程调度，在内存这个维度上把性能做到了极致。
